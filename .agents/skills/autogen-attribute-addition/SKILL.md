# Supporting a New Attribute in an Autogen Resource

When the Atlas Admin API adds a field, the scheduled codegen bot surfaces it in a
"DO NOT MERGE" visibility PR for production resources. Incorporating it requires
complementary work that the bot does not do. Follow these steps; each one defers
to the relevant specialized skill where one exists.

## 1. Understand the change

- Read the visibility PR diff: which generated files changed, and which endpoint's
  schema gained the field. Spec files often show only an `x-xgen-sha` bump because
  the spec content already landed on master via the internal-resources bot PR.
- The resource, singular data source, and plural data source are generated from
  *different endpoints*. A field appearing in only one of them usually means only
  that endpoint's response schema changed — check the others before assuming a gap.
- Trace the origin in the `mms` repo (`~/mongodb/mms`): `git log -S "<field description>"`
  on the view class, or grep for the OpenAPI schema name. The commit message gives the
  owning CLOUDP ticket and team — useful context for the ticket and for reporting
  spec issues upstream.

## 2. Create the Jira ticket

Create a CLOUDP ticket before starting implementation. Conventions: summary prefixed
`[TF]`, type Story, priority Major - P3, component `Terraform`, Assigned Teams
`APIx DevOps Integrations`. Describe the API change origin, the affected
resource/data sources, and the scope of work. Branch name should start with the
ticket key (e.g. `CLOUDP-413373-...`) so Jira links the work automatically.

## 3. Check schema consistency before regenerating

Compare the new field's spec schema with sibling schemas already exposed on the
resource or other data sources:

- **Collection type**: if siblings expose the field as a Set but the new schema lacks
  `uniqueItems: true`, codegen emits a List. Align with a type override in
  `tools/codegen/config.yml`:

  ```yaml
  datasources:
    schema:
      overrides:
        roles:
          type: set # Align with the resource and singular data source
  ```

- **Override key semantics**: override keys are matched after stripping the
  `results.` prefix from the attribute path, so one key applies to *both* the
  singular data source attribute and the plural `results.*` attribute.
  `results.`-prefixed override keys are silently ignored — do not use them.
  (`ignores` and `aliases` are the opposite: they require explicit `results.`-prefixed
  entries.) Type overrides are no-ops when the attribute already has the target type.
- **Descriptions**: each surface inherits wording from its own endpoint schema, so the
  same field can carry different descriptions. Prefer reporting wording inconsistencies
  (and missing `uniqueItems`) to the owning API team over papering them with local
  description overrides — the provider then picks up the fix on a future regen.

## 4. Regenerate

```bash
make autogen-update-api-spec        # fetch + flatten latest prod spec
make autogen-generate-resources resource_name=<name>
go build ./...
```

Verify the diff only touches the expected generated files (schema, model yaml, spec
SHA). If model generation fails on an override, read the error — it usually means the
override hit an attribute on another surface (see step 3).

## 5. Acceptance tests

Extend the existing consolidated test (resource + singular DS + plural DS share one
test function) rather than adding a new one. Assert the new attribute in the plural
`results` checks. Follow the `acceptance-test-patterns` skill for ordering rules —
index-based `results.N` checks are acceptable only when every result is guaranteed
equivalent or unique within the test. If credentials are not available locally, state
that CI will provide the first real run.

## 6. Docs and changelog

- Regenerate docs: `scripts/generate-doc.sh <resource_name>` (requires templates in
  `templates/`). Only commit the generated doc for the affected resource.
- Add `.changelog/<PR number>.txt` with a `release-note:enhancement` block per
  affected resource/data source (see existing entries for format). The filename must
  match the PR number — predict it from the latest issue/PR number and verify right
  after creating the PR.

## 7. PR

Follow the `pr-and-documentation-standards` skill and the PR template. Reference the
visibility PR and the CLOUDP ticket in the description, and note any upstream spec
issues (missing `uniqueItems`, inconsistent descriptions) reported to the owning team.
