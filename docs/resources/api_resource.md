# Resource: mongodbatlas_api_resource

`mongodbatlas_api_resource` is a generic Terraform resource that wraps an arbitrary Atlas Admin API endpoint. Use it when:

1. The endpoint is in **preview** (`application/vnd.atlas.preview+json`) and you need Day-0 access.
2. The endpoint is GA but does not yet have a hand-crafted typed resource.

For long-term use, prefer a typed resource as soon as one is available â€” typed resources offer richer validation, better plan output, and stronger schema guarantees.

~> **IMPORTANT â€” `output` is NOT marked Sensitive.** The full API response is stored in the `output` Dynamic attribute and surfaces in `terraform plan` / `terraform apply` / `terraform show`. If the endpoint returns secret values (for example a one-shot API key on Create), pipe them through your own [`output { sensitive = true }`](https://developer.hashicorp.com/terraform/language/values/outputs#sensitive-suppressing-values-in-cli-output) block. This trade-off favours debuggability and matches the convention used by [`azapi_resource`](https://registry.terraform.io/providers/Azure/azapi/latest/docs/resources/resource).

## Example Usage

Minimal example against the service-accounts endpoint. This is the proof-of-life exercised by the resource's acceptance test.

```terraform
# Minimal example: manage an Atlas Service Account via the generic resource.
# A typed `mongodbatlas_service_account` resource exists â€” prefer it for production.

resource "mongodbatlas_api_resource" "this" {
  path                  = "/api/atlas/v2/orgs/${var.org_id}/serviceAccounts"
  id_attribute          = ["clientId"]
  create_only_body_keys = ["secretExpiresAfterHours"]

  body = {
    name                    = "example-generic-sa"
    description             = "Service account managed via mongodbatlas_api_resource"
    roles                   = ["ORG_READ_ONLY"]
    secretExpiresAfterHours = 24
  }
}

data "mongodbatlas_api_resource" "this" {
  path = mongodbatlas_api_resource.this.id
}

output "service_account_client_id" {
  value = mongodbatlas_api_resource.this.output.clientId
}

output "service_account_first_secret" {
  description = "Returned only on Create. After the first refresh this is null."
  value       = try(mongodbatlas_api_resource.this.output.secrets[0].secret, null)
  sensitive   = true
}
```

## Schema

### Required

- `path` (String) Collection path used for Create (e.g. `/api/atlas/v2/orgs/<orgId>/serviceAccounts`). For singleton endpoints, also used as-is for Read/Update/Delete. For collection endpoints, the provider appends `/<id>` from `output` (see `id_attribute`) for Read/Update/Delete.

### Optional

- `body` (Dynamic) Request body sent on Create/Update. The reshape engine drives drift detection: keys you declare here are tracked; fields Atlas adds server-side are stored in `output`.
- `create_method` (String) HTTP method for Create. One of POST, PUT, PATCH. Defaults to POST.
- `create_only_body_keys` (Set of String) Top-level body keys that the endpoint accepts only on Create and rejects on Update (for example `secretExpiresAfterHours`). These keys are stripped from the payload before the Update request is issued.
- `id_attribute` (List of String) Response field name(s) that identify the resource. The provider reads each name from `output` and appends `/<value>` to `path` for Read/Update/Delete (in order, for composite IDs). Omit for singleton endpoints where Create/Read/Update/Delete share the same path. Example: `["clientId"]` for service accounts; `["databaseName", "username"]` for database users.
- `preview` (Boolean) Shorthand for `version_header = "application/vnd.atlas.preview+json"`. Mutually exclusive with `version_header`.
- `sensitive_body` (Dynamic, Sensitive) Sensitive fragment merged into the request body at Create/Update. Values are never written to state; their keys are excluded from drift comparison.
- `update_method` (String) HTTP method for Update. One of PATCH, PUT, POST. Defaults to PATCH.
- `version_header` (String) Atlas API version media type used for both Accept and Content-Type headers. When unset, defaults to today's UTC date in the form `application/vnd.atlas.<YYYY-MM-DD>+json` (Atlas snaps the date down to the latest published version on or before it). The resolved value is persisted at Create time and stays stable for the lifetime of the resource. Mutually exclusive with `preview`.

### Read-Only

- `id` (String) Synthetic identifier set to the resolved read URL.
- `output` (Dynamic) Full API response from the most recent operation. Access fields with `mongodbatlas_api_resource.<name>.output.<field>`. NOT marked Sensitive â€” see the resource-level warning.

## Path derivation

`path` is the Create endpoint (the collection path, or the singleton path). For Read/Update/Delete the provider appends `/<value>` for each entry in `id_attribute`, reading values from the API response (`output`).

| `id_attribute` | Read/Update/Delete URL |
|----------------|------------------------|
| omitted | `path` (singleton â€” same path for all operations) |
| `["clientId"]` | `path` + `/${output.clientId}` |
| `["databaseName", "username"]` | `path` + `/${output.databaseName}/${output.username}` |

Use native HCL interpolation (`"${var.org_id}"`) for variables in `path`. There is no provider-level path interpolation.

## API version â€” `version_header` and `preview`

When `version_header` is unset, the resource captures today's UTC date at Create (e.g. `application/vnd.atlas.2026-05-11+json`) and persists it in state. Atlas snaps the date down to the latest published API version on or before it, so this effectively means "use whatever was latest when this resource was first created" â€” newer resources get newer versions automatically, existing resources stay stable for their lifetime.

Two ways to override:

- Set `preview = true` to opt into the preview channel (`application/vnd.atlas.preview+json`). Use for endpoints that only respond on the preview channel.
- Set `version_header` explicitly to pin a specific Atlas API version date â€” useful if you need to lock behavior to a known release.

The two attributes are mutually exclusive â€” configuring both raises a validation error.

## Input secrets â€” `sensitive_body`

`sensitive_body` is merged into the request body at Create/Update time, never written to state, and excluded from the reshape comparison so the API's redacted echo (`"value": "<redacted>"`) does not show as drift. Typical use is for fields like third-party API tokens, OTLP header values, or any practitioner-supplied secret the API stores but redacts on read.

```hcl
resource "mongodbatlas_api_resource" "metric_integration" {
  path         = "/api/atlas/v2/groups/${var.project_id}/metricIntegrations"
  id_attribute = ["metricIntegrationId"]
  preview      = true

  body = {
    endpoint        = "https://otlp.example.com"
    integrationType = "OTEL"
    providerType    = "CUSTOM"
    headers = [
      { name = "X-API-Token" }, # value comes from sensitive_body below
    ]
  }

  sensitive_body = {
    headers = [
      { value = var.otlp_token },
    ]
  }
}
```

## Drift detection rules

The reshape engine that updates `body` after every Read follows these rules:

- **Configured key present in response** â†’ take the response value. Differs from configured â†’ drift.
- **Configured key absent from response** â†’ preserve the configured value. Handles write-only request fields (for example `secretExpiresAfterHours`) that the API never echoes back.
- **Configured key present but `null` in response** â†’ write `null` in state. Treated as real drift.
- **Response key not present in configured body** â†’ dropped. Server-only fields stay in `output`.
- **Scalar arrays that contain the same elements in a different order** â†’ configured order preserved. Set-like fields such as `roles` or `labels` do not cause reorder diffs.
- **Caveat.** If an endpoint *omits* unset optional fields from its response (rather than returning `null`), an out-of-band *unset* of that field cannot be distinguished from "the API doesn't echo it back" and is silently masked. Out-of-band value *changes* are still detected. Most Atlas endpoints return explicit `null`, so this rarely bites.

## One-shot output secrets

Some endpoints return a secret only in the Create response (`secrets[0].secret` on service accounts, `secret` on AI Model API Keys). Lifecycle:

1. Capture the value at apply time through a sensitive Terraform output.
2. On the next refresh the field becomes `null` in `output`. Because `output` is fully Computed, this does **not** produce a non-empty plan.
3. There is no recovery path â€” recreate the resource if you lose the captured value.

## Import

Import an existing endpoint by setting the import ID to the resolved read URL:

```
$ terraform import mongodbatlas_api_resource.example /api/atlas/v2/orgs/<ORG_ID>/serviceAccounts/<CLIENT_ID>
```

After import, re-declare `body` and (if applicable) `sensitive_body` in HCL. The next plan will surface drift until your configured `body` matches what Atlas returns.

## Limitations

The resource is intentionally lean. The following may land in future versions:

- **No async polling.** Synchronous CRUD only â€” endpoints that 202-Accept and require polling are not yet supported.
- **No schema validation.** All semantic validation is deferred to Atlas; errors surface verbatim in diagnostics.
- **No opt-in response projection.** The entire response lives in `output`. A JMESPath / dotted-path projection (`response_export_values`) is on the roadmap if state size becomes a concern.
- **No support for endpoints whose Create path differs from `path` by more than a trailing `/<id>` segment** (for example custom verbs like `:invite`) or whose Update path differs from the Read path. Use a typed resource for these.
